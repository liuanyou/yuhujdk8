### Root Cause
When deoptimization is triggered in a Yuhu-compiled method, the HotSpot deoptimization system tries to:
1. Create a `compiledVFrame` for the deoptimizing frame
2. Look up the `ScopeDesc` (scope descriptor) at the current PC
3. **FAILS** because Yuhu-compiled nmethods don't have proper scope descriptors

The deoptimization system requires scope descriptors to:
- Reconstruct interpreter frames from compiled frames
- Map registers and stack slots back to bytecode locals and stack
- Handle multi-level deoptimization (inlined methods)

## Why Yuhu Lacks Scope Descriptors

Yuhu uses LLVM for code generation, which doesn't integrate with HotSpot's scope descriptor mechanism. The scope descriptors are normally generated by C1/C2 compilers during code generation to record:
- PC-to-bytecode BCI (Bytecode Index) mappings
- Register/stack slot allocation maps
- Inlining information
- OopMap (GC safe point) information

## Stub-Related Aspect

The static call stub mechanism **compounds this problem** because:

1. **Return Address Issue**: When calling through a stub:
   ```
   Compiled Code → Static Call Stub → _from_compiled_entry → Target Method
   ```
   The return address on the stack points into the stub or the caller's compiled code.

2. **PC Lookup Failure**: During deoptimization, the system uses the return PC to:
   - Find the containing nmethod
   - Look up the scope descriptor at that PC
   - But if the PC is in a stub (which has no scope descriptors), lookup fails

3. **Frame Construction**: Even if the nmethod is found, the PC might not match any recorded scope descriptor entry, causing the `guarantee(pd != NULL)` failure.

## Immediate Solution Options

### Option 1: Disable Deoptimization for Yuhu Methods (Workaround)
Modify Yuhu nmethod creation to mark methods as non-deoptimizable.

**Pros**: 
- Quick fix
- Avoids the missing scope descriptor issue

**Cons**:
- Prevents proper exception handling
- Breaks uncommon trap mechanism
- May cause incorrect behavior for some Java semantics

### Option 2: Generate Minimal Scope Descriptors
Add minimal scope descriptor generation to Yuhu compiler.

**Pros**:
- Enables proper deoptimization
- Maintains Java semantics

**Cons**:
- Significant engineering effort
- Requires deep integration between LLVM and HotSpot metadata

### Option 3: Use Inline Assembly Instead of Stubs for Direct Calls
Generate inline assembly directly in the LLVM IR for Method* handling.

**Pros**:
- Eliminates stub-related PC issues
- Keeps PC within the nmethod where scope descriptors exist

**Cons**:
- Still requires scope descriptors to be generated
- Requires LLVM inline assembly generation

## Recommended Immediate Action

Since we're using stubs and encountering deoptimization issues, the recommended approach is:

### Step 1: Add Minimal Scope Descriptor Support
Generate a simple scope descriptor for each Yuhu-compiled method that:
- Maps the entry PC to BCI 0
- Maps any potential deopt points to their corresponding BCI
- Provides basic OopMap information

### Step 2: Ensure Stub Return Addresses Are Handled
Modify the static call stub to ensure that:
- The return address points back into the caller's nmethod (not the stub)
- The stub is completely transparent to the deoptimization system

### Step 3: Mark Deopt-Unsafe Regions
For regions where scope descriptors cannot be accurately generated:
- Disable uncommon traps
- Ensure no safepoints are inserted by LLVM

## Implementation Plan

### Phase 1: Basic Scope Descriptor Generation
1. Create `YuhuScopeDescRecorder` class (similar to C1/C2's DebugInformationRecorder)
2. Record at minimum:
   - Method entry point → BCI 0
   - Method exit points → BCI (end of method)
   - Exception handler entries → corresponding BCIs
3. Attach scope descriptors to the nmethod during compilation

### Phase 2: Stub Transparency
1. Ensure static call stubs use proper calling conventions
2. Verify return addresses point into caller nmethod
3. Test deoptimization through stub calls

### Phase 3: Full Deopt Support
1. Add LLVM instrumentation to track BCI mappings
2. Generate OopMaps for GC-safe points
3. Support inlined method deoptimization
