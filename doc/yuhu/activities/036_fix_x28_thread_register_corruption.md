### 036 Fix x28 Thread Register Corruption in Deoptimization

#### 1. Background

On AArch64 HotSpot, register `x28` is reserved as the **JavaThread* thread register**. The Yuhu compiler generates AArch64 code for methods such as `sun.nio.cs.UTF_8$Encoder::encodeArrayLoop`, and deoptimization is handled by the shared `DeoptimizationBlob` (the same infrastructure used by C1/C2).

During testing of a Yuhu-compiled `encodeArrayLoop`, the JVM crashed in the deoptimization path:

- Crash point: `HandleMarkCleaner::HandleMarkCleaner(Thread*)`
- Caller: `Deoptimization::fetch_unroll_info(JavaThread*)`
- Symptom: `thread->_last_handle_mark` contained the invalid value `0x24`
- Effect: `last_handle_mark()->push()` dereferenced an invalid pointer and triggered a fatal error.

The key observation was: **at the entry of `fetch_unroll_info`, the incoming `JavaThread` already had `_last_handle_mark == 0x24`**, so the corruption happened **before** deoptimization C++ code ran.

#### 2. Where x28 gets corrupted

Further debugging of Yuhu-generated code for `encodeArrayLoop` showed:

- The reverse-deopt entry for the Yuhu-compiled method is in the deopt blob at address, e.g., `0x000000010ca12d4c`.
- Before we reach the deopt blob, execution runs inside the Yuhu-compiled `encodeArrayLoop`.
- Inside that method, there is an indirect call of the form:

```asm
... ; Yuhu compiled body
blr x8   ; call another helper/method via x8
...
```

At the point of this `blr x8`, **`x28` had already been clobbered** by Yuhu-generated code or by the callee, and no code restored it afterwards. Since HotSpot expects `x28` to be preserved as the thread register across Java execution, this violates the ABI/VM convention.

When deoptimization is triggered and control eventually reaches the deopt blob, it assumes:

- `x28` still holds a valid `JavaThread*`

But due to Yuhu’s code generation, `x28` instead holds garbage. Thus the deopt blob’s store:

```asm
str x8, [x28, #0x200]
```

writes `_last_handle_mark` to an invalid location, which manifests as `0x24` in our specific crash.

**Root cause:**

> Yuhu-generated code does **not preserve register `x28`**, even though HotSpot uses `x28` as the dedicated JavaThread register. As a result, the thread’s `_last_handle_mark` field is overwritten with an invalid value when the deopt blob runs, causing `HandleMarkCleaner` to crash.

#### 3. Constraints and design requirements

From HotSpot’s AArch64 conventions and Yuhu’s design we get the following hard requirements:

- `x28` must always point to the current `JavaThread` during Java execution.
- Deoptimization, safepoints, and runtime calls all assume a valid `JavaThread*` in `x28`.
- Yuhu must either:
  - Treat `x28` as a **reserved, non-allocatable register**, or
  - Explicitly **save and restore** `x28` around any code that might clobber it.

Additionally, per our internal practice guideline **“Preserve critical registers for inline assembly safety”**, we already consider `x20`, `x27`, and `x28` as critical and should ensure they are not silently destroyed by LLVM-generated code.

#### 4. Solution options considered

We considered several approaches to fix this without immediately changing HotSpot itself.

##### Option A: Save/restore x28 around every outbound call

At each call generated by Yuhu (both Java calls and VM/runtime calls), emit:

```asm
stp x28, xzr, [sp, #-16]!
blr x8               ; or call target
ldp x28, xzr, [sp], #16
``

**Pros:**
- Simple to implement on the Yuhu side.
- Does not require touching LLVM backend internals.
- Makes the preservation explicit and local to the call sites.

**Cons:**
- Adds stack traffic for every call, with potential performance impact.
- All call sites must consistently use the wrapped calling sequence.

##### Option B: Save x28 in the function prologue and restore in epilogue only

In the Yuhu-generated function prologue:

```asm
stp x28, x29, [sp, #-16]!
mov x29, sp
; ... normal prologue ...
```

And in the epilogue:

```asm
; ... normal epilogue ...
ldp x28, x29, [sp], #16
ret
```

Plus a **strict rule** that Yuhu’s code generator and LLVM must not use `x28` as an allocatable register inside the function body.

**Pros:**
- Only two save/restore operations per function (not per call).
- Cleaner from an ABI perspective: `x28` is effectively treated as a callee-saved, special-purpose register.

**Cons:**
- Requires configuring LLVM so that it never allocates `x28` for temporaries or spills.
- If LLVM is not informed about this constraint, it may still clobber `x28` in the body.

##### Option C: Model JavaThread* as an explicit SSA value and back it with x28 via inline asm constraints

Expose the thread pointer as a function parameter in LLVM IR and keep it as a first-class value. Use inline-asm or backend constraints to force that value to live in `x28` at the ABI boundaries.

**Pros:**
- Architecturally clean: the thread pointer is explicit in the IR.
- Could enable more flexible transformations in the future.

**Cons:**
- Significantly more complex to implement.
- Requires deep integration with LLVM’s register allocator.

#### 5. Chosen direction

For this issue, the pragmatic direction is:

1. **Treat `x28` as a critical reserved register in Yuhu-generated code.**
2. In the short term, enforce preservation via **explicit save/restore** at well-defined boundaries:
   - At least around indirect calls (`blr x8`) that may jump to arbitrary code.
   - Optionally also at other outbound call sites.
3. Longer term, investigate tightening LLVM’s register allocation so that `x28` is marked non-allocatable in the AArch64 subtarget used by Yuhu, enabling the simpler prologue/epilogue-only preservation (Option B).

This respects the existing HotSpot convention that `x28` holds `JavaThread*` and avoids changing any HotSpot runtime or deopt blob code.

#### 6. Alternative solution: Access x28 via x29 frame pointer

The previous approach of saving x28 in the Yuhu stack frame proved flawed because we cannot guarantee that `CreateReadThreadRegister()` reads the initial x28 value at function entry. The LLVM-generated prologue runs first, potentially overwriting x28 before Yuhu code executes.

The new approach leverages the fact that **LLVM-generated AArch64 prologue saves x28 to the stack** as a callee-saved register. We can access the original x28 value through the frame pointer (x29):

1. LLVM generates prologue that saves x28: `stp x28, x27, [sp, #-96]!`
2. LLVM also saves x29 (frame pointer): `stp x29, x30, [sp, #80]`
3. Then sets x29: `add x29, sp, #0x50`
4. So x28's saved value is at a predictable offset from x29

#### 7. Implementation approach

The fix will be implemented in the `YuhuTopLevelBlock::get_direct_callee()` method, which generates hand-written assembly code. The approach:

1. **Generate placeholder instruction**: Insert a placeholder instruction during initial code generation
2. **Analyze actual prologue**: After code generation, use a prologue analyzer similar to `YuhuPrologueAnalyzer::analyze_prologue_stack_bytes()` to scan the actual function prologue
3. **Calculate x28 offset**: Determine the exact offset of x28's saved value relative to x29
4. **Patch the placeholder**: Replace the placeholder with the correct `ldur x8, [x29, #offset]` instruction

This approach handles potential variations across different LLVM versions by dynamically analyzing the generated prologue rather than assuming fixed offsets.

#### 8. Summary

This activity documents the root cause analysis and solution for x28 thread register corruption in Yuhu-compiled code, which led to crashes during deoptimization and safepoint operations.

The core issue was that Yuhu-generated code did not preserve the x28 register (which should always hold the JavaThread* on AArch64), causing it to be corrupted during execution. When deoptimization occurred, the deoptimization blob expected a valid JavaThread* in x28 but found garbage, leading to crashes in HandleMarkCleaner.

The solution involves accessing the original x28 value (saved by LLVM's prologue) through the frame pointer (x29) using dynamic prologue analysis to determine the correct offset.
